# ADR-0007 人事输入默认值与最简场景

> 人事变更记录改为可选，提供默认在职区间、默认工种、业务线可空，兜底最简使用场景。

## 状态

accepted

## 上下文（Context）

- ADR-0003 D2 把人事变更记录（onboard/leave/transfer）作为必填输入，用于动态构建全员。
- 但最简场景下用户不想录变更记录：全员稳定、无入职离职转线，录变更记录是纯负担。
- 需要默认值兜底，让"只设区间 + 录项目 + 给员工名单"即可生成。
- 业务线在某些场景无意义，不应强制。
- 工种是项目工种覆盖约束的依赖项，但员工侧可给默认值。

## 决策（Decision）

### D1 人事变更记录改为可选

- `StaffChangeRecord` 不再是必填输入。
- 未提供任何变更记录的员工，按默认在职区间与默认工种兜底（见 D2/D3）。
- 提供了变更记录的员工，按变更记录推导在职区间/业务线/工种（ADR-0003 D2 不变）。
- 混合支持：同一会话内部分员工带变更记录、部分用默认值。

### D2 默认在职区间 = 全局生成区间

- 用户在会话开始时设定一个**全局生成区间**（start_date, end_date）。
- 未提供变更记录的员工，视为在该全局区间内一直在职。
- 分母计算：全员可用工时按该全局区间内的有效工作日 × 8h 计算（对用默认值的员工）。
- 提供了 onboard/leave 的员工，在职区间按记录推导，可能短于全局区间。

### D3 默认工种 = "研发人员"

- 员工未提供工种时，归入默认工种"研发人员"。
- 项目 `required_job_types` 仍需显式声明（项目侧无默认）。
- 若项目 required_job_types 含"研发人员"，用默认工种的员工可满足覆盖；否则需用户提供对应工种员工。

### D4 业务线可空

- 员工未提供业务线时，业务线为空（不适用）。
- 业务线不参与任何生成约束（工种覆盖、比例、工时分配均不依赖业务线）。
- 业务线仅作为记录维度（可用于导出汇总的分组，若用户需要）。
- 提供了 transfer 记录的员工，业务线按记录分段；未提供则为空。

### D5 最简场景的最小输入

最简使用场景下，用户只需提供：
1. 全局生成区间（start_date, end_date）。
2. 员工名单（仅姓名/标识，甚至可不带工种列，默认"研发人员"）。
3. 项目（含周期比例、所需工种、关联员工）。

无需任何人事变更记录、无需业务线、无需年假额度（年假默认 0？见开放问题）。

## 后果（Consequences）

- 正面：
  - 最简场景门槛大幅降低，用户可跳过人事变更录入。
  - 默认值让小规模/稳定团队场景开箱即用。
  - 业务线退化为可选记录维度，不强加约束。
- 负面：
  - 数据模型需支持"在职区间默认值"与"业务线可空"。
  - 分母计算需区分"默认在职员工"与"带变更记录员工"的在职区间。
  - 默认工种"研发人员"是固化常量，若用户项目工种不含研发，默认值员工无法满足覆盖，需用户显式提供工种。
- 后续需要做的事：
  - 确认年假默认值（0？还是也可选不休年假？）。
  - 数据模型与生成器支持默认值兜底逻辑。
  - UI 最简流程：全局区间 + 员工名单 + 项目即可启动生成。

## 备选方案（Alternatives）

- 变更记录必填不变：被否，最简场景负担过重。
- 默认在职区间跟项目区间：被否，多项目时分母计算复杂且不一致，用户选全局区间。
- 默认业务线固定字符串：被否，业务线可空更灵活。
- 默认工种为空：被否，项目工种覆盖约束需工种匹配，默认"研发人员"让最简场景可生成。
- 项目 required_job_types 也给默认：被否，项目工种需求是业务输入，不应默认。

## 修订记录

- 2026-07-17：业务线定位澄清。业务线与项目有关（项目有业务线归属），会造成人员排布的倾向性（软偏好，非硬约束）。**当前不作考虑，不实现**。业务线字段在数据模型保留为预留，MVP 阶段可空、不录入、不消费。未来启用时：项目有 business_line 字段，生成时某项目倾向用同业务线员工（软偏好，不影响工种覆盖等硬约束）。
