# ADR-0005 多项目全局生成与字典管理

> 同期多项目一次性全局生成（统一调度员工工时）；工种与业务线作为预置字典管理；人事数据支持 CSV/Excel 导入。

## 状态

accepted

## 上下文（Context）

- 同一时间段常有多个项目并行，同一员工可能在多项目间分配工时。
- 硬约束：单员工单日跨项目工时求和 = 8h（满载，可拆分，FR-003、ADR-0008）。
- 人事数据规模可达几十至上百人，跨数年变动；逐条 UI 录入负担重。
- 工种与业务线是跨项目、跨员工复用的分类维度，重复声明易不一致。
- 节假日 API（timor.tech）支持历史年份查询，补录去年/前年项目可行。

## 决策（Decision）

### D1 多项目全局一起生成

- 用户一次提交**所有同期项目**（同一生成批次），生成器统一调度员工工时。
- 生成器在单次运行内感知全部项目的工时分配，天然满足"单员工单日跨项目求和 = 8h"。
- 不支持"逐个生成 + 跨项目冲突检测"的增量模式；参数调整需全量重生成该批次。
- 生成批次 = 用户本次提交的全部项目集合；不同批次（不同时间段）可独立生成互不影响。

### D2 工种与业务线预置字典管理

- `JobType` 与 `BusinessLine` 作为独立字典实体维护，有 id 与 name。
- 项目录入 required_job_types 时从字典引用；员工 onboard 时从字典引用 job_type / business_line。
- 字典支持新增、重命名（重命名不影响已生成记录，因记录引用 id）。
- 字典不提供删除（避免悬挂引用）；若需停用，后续可加"停用"标记（当前不做）。

### D3 人事数据双模式录入

支持两种模式，用户按规模自选：

**模式 A：全量导入**
- CSV/Excel 导入员工列表 + 人事变更记录（统一格式模板）。
- 适合首次初始化或大规模批量补录。

**模式 B：混合模式**
- CSV/Excel 导入员工列表（基础信息）。
- 人事变更记录在 UI 中逐条编辑（时间线表格）。
- UI 编辑结果可导出（CSV/Excel），以便后续复用或版本归档。

- 导入格式模板需固化并文档化（列定义、日期格式、枚举值）。
- 导入时校验：字段完整、日期合法、onboard/leave/transfer 逻辑一致、工种/业务线引用字典存在。

### D4 节假日支持历史年份

- 节假日 API 客户端支持按任意年份（历史/当前/未来）获取并缓存。
- 补录历史项目时按需获取历史年份节假日。
- 缓存按年度组织，历史年份一旦缓存可离线复用。
- 约束：若 timor.tech 对某历史年份无数据，报错并提示用户手动提供（降级路径待定）。

## 后果（Consequences）

- 正面：
  - 全局生成让跨项目 = 8h 约束在算法内自然满足，无需事后校验或冲突检测。
  - 字典管理保证工种/业务线一致性，重命名安全。
  - 双模式录入适配小规模 UI 与大规模导入两种场景。
  - 历史年份支持让补录场景完整可用。
- 负面：
  - 全局生成在项目数量多时单次计算量大；需验证 NFR-010（≤5s）在多项目下的可行性，可能需放宽或优化。
  - 全量重生成意味着小调整也需重跑全部，用户体验需靠"生成快"弥补。
  - 字典管理增加一层实体维护 UI。
  - 导入格式模板需维护与文档化，格式错误处理增加复杂度。
- 后续需要做的事：
  - 压测多项目全局生成的性能，确认或修订 NFR-010。
  - 定义导入格式模板（列、类型、枚举）并写入文档。
  - 设计字典管理 UI（新增/重命名）。
  - 验证 timor.tech 历史年份覆盖范围，定义无数据时的降级策略。

## 备选方案（Alternatives）

- 逐个生成 + 冲突检测（方案 B）：被否，跨项目状态管理复杂，易引入不一致。
- 逐个生成 + 事后合并校验（方案 C）：被否，易失败、多次重试，体验差。
- 工种/业务线临时声明：被否，跨项目跨员工复用场景下一致性难保证。
- 仅 UI 逐条录入人事数据：被否，大规模场景负担过重。
- 仅 CSV 导入：被否，小规模或变更调整时 UI 更灵活，应并存。
- 节假日仅当前/未来：被否，补录历史项目是核心场景之一。
