# ADR-0003 比例口径与人事变更记录

> 钉死投入比例的分母语义（周期内全员可用工时），并引入"人事变更记录"作为生成器的动态全员构成输入。

## 状态

accepted

## 上下文（Context）

- ADR-0002 D1 遗留开放问题：`target_ratio` 的分母是"关联员工"还是"全局员工池"？
- 业务事实：人员会入职、离职、切换业务线；"全员"不是静态集合，而是随日期变化。
- 项目投入比例按**周期**（年/季度/月）表达，不是全期单一比例。
- 项目仍关联特定员工子集（须覆盖所需工种），但比例分母用全员，语义为"该项目占用全员工时的比例"。

## 决策（Decision）

### D1 比例口径：周期内全员可用工时为分母

- 项目录入时声明**周期粒度**：`year` | `quarter` | `month`。
- 每个周期录入一个 `target_ratio`（可按周期不同）。
- 分母 = 该周期内**全员**可用工时 = 全员在该周期内的有效工作日 × 8h（有效工作日 = 周期内日期 − 节假日 − 年假日，且员工在职）。
- 分子 = 项目在该周期内实际分配的工时（生成结果）。
- 约束：生成后每个周期 `Σ项目分配工时 / 全员可用工时 ≈ target_ratio`（容差待定）。
- 全员构成由人事变更记录按日期动态决定（见 D2）。

### D2 人事变更记录作为输入

新增输入实体 `StaffChangeRecord`，类型分三种：

| 类型 | 字段 | 对生成的影响 |
|---|---|---|
| `onboard` 入职 | person_id, date, job_type, business_line | 该日期起计入全员；此前不计入 |
| `leave` 离职 | person_id, date | 该日期起退出全员；此后不分配工时 |
| `transfer` 业务线切换 | person_id, date, new_business_line | 该日期起业务线变更；工种不变 |

- 员工的"在职区间"由首条 onboard 与（若有）leave 记录界定。
- 业务线归属按 transfer 记录按日期分段。
- 工种在 onboard 时确定；本工具不支持工种变更（若需要，另起 ADR）。
- 生成器在计算分母与分配工时时，按日期查询当日全员构成。

### D3 项目关联员工与全员的区分

- 项目录入时仍需指定**关联员工子集**（满足 required_job_types 覆盖）。
- 关联员工是"可被分配到该项目的人"；全员是"比例分母的计算范围"。
- 关联员工必须 ⊆ 当日全员且在职。
- 一个员工的在职区间若与项目区间无交集，不能作为关联员工。

## 后果（Consequences）

- 正面：
  - 比例口径贴近真实审计语义（项目占全员工时比例）。
  - 人事变更记录让"全员"按日期动态构成，准确反映入职/离职/转线。
  - 周期粒度支持按年/季/月差异化投入强度。
- 负面：
  - 分母计算复杂度上升：需按周期逐日聚合全员在职状态与可用工作日。
  - 人事变更记录是新的必填输入，录入负担增加。
  - 周期边界与项目区间边界可能不对齐（项目跨多周期），需明确周期内项目天数的归属规则。
  - 关联员工 ⊆ 全员的校验在动态全员下更复杂。
- 后续需要做的事：
  - 定义周期内项目天数归属规则（项目跨周期时如何拆分）。
  - 定义比例换算容差（生成结果与目标比例的允许偏差）。
  - 设计人事变更记录的 UI 录入与校验。

## 备选方案（Alternatives）

- 分母用关联员工（ADR-0002 原推断）：被否，用户明确为全员。
- 比例全期单一值不按周期：被否，投入强度随周期变化。
- 全员静态（不引入人事变更记录）：被否，无法表达入职/离职/转线，分母失真。
- 工种也可变更：暂不支持，避免复杂度；若需要另起 ADR。
