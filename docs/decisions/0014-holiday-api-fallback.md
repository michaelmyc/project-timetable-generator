# ADR-0014 节假日 API 降级策略

> timor.tech API 重试 3 次失败后，降级为"仅周末排除"模式（不考虑法定节假日与调休）；降级时 UI 提示用户；不内置静态表、不做手动导入。

## 状态

accepted

## 上下文（Context）

- timor.tech 是第三方免费节假日 API，无 SLA，可能宕机/限流/历史年份缺失。
- 原文档（ADR-0002 D4、ADR-0005 D4）写"无数据时降级策略待定"，一直未闭环。
- 节假日是合规硬约束（节假日无工时），无节假日数据原本意味着工具不可用——单点依赖。
- 用户决策：重试 3 次失败后降级为"只考虑周末"；需 UI 提示避免用户无感等待；不维护内置静态表；不做手动导入。

## 决策（Decision）

### D1 三层尝试：API → 重试 → 降级

1. 首选调用 timor.tech API 获取该年度节假日（含调休工作日标记）。
2. 失败时重试，**最多 3 次**。
3. 3 次都失败 → 降级为"仅周末排除"模式，不终止生成。

### D2 降级模式：仅周末排除

- 降级时不查 Holiday 数据，直接按日期的星期几判定：
  - 周六、周日 = 非工作日（无工时）。
  - 周一至周五 = 工作日（可分配工时）。
- **不考虑法定节假日**（如国庆、春节仍按工作日）。
- **不考虑调休工作日**（如周末补班仍按非工作日）。
- 降级模式生成结果可能不符合法定节假日合规要求——由用户自行判断是否接受。

### D3 降级时 UI 提示

- API 失败并降级时，UI 明确提示用户：
  - "节假日数据获取失败（timor.tech 不可用）"
  - "已降级为仅周末排除模式，生成结果可能不符合法定节假日"
  - "如需精确节假日，请检查网络后重试"
- 提示时机：生成开始前（节假日获取阶段失败时立即提示），不是生成完成后才说。
- 与 ADR-0009 D5 进度反馈一致：进度阶段包含"节假日获取"，失败时显示降级状态。

### D4 不内置静态表、不做手动导入

- 不维护内置节假日静态表（用户明确不要）。
- 不做手动导入节假日 CSV 的降级路径（用户明确不要）。
- 降级只有"仅周末排除"一种模式，简单直接。

### D5 缓存仍是常态优化

- API 成功时，结果仍本地缓存（ADR-0006 D3、ADR-0002 D4），后续优先读缓存。
- 缓存命中时不调用 API，不触发降级。
- 降级仅发生在"未缓存 + API 3 次失败"时。

## 后果（Consequences）

- 正面：
  - timor.tech 不再是单点——API 失败不阻塞生成，工具始终可用。
  - 降级策略简单（仅周末排除），无额外数据维护负担。
  - UI 提示让用户知情，避免无感等待或不知情使用不合规结果。
  - 缓存命中时无 API 调用，常态体验不受影响。
- 负面：
  - 降级模式生成结果可能不符合法定节假日（如国庆周仍分配工时），合规性降低。
  - 用户需自行判断降级结果是否可接受——工具不强制阻断。
  - 调休工作日在降级时不被识别（周末补班仍按非工作日），可能少分配工时。
- 后续需要做的事：
  - 修订 FR-004/FR-011 的"离线报错"措辞为"降级 + UI 提示"。
  - 实现重试 3 次 + 降级逻辑。
  - 实现降级 UI 提示（生成进度阶段）。
  - 验证 timor.tech 历史年份覆盖（非阻塞，了解即可）。

## 备选方案（Alternatives）

- 内置静态节假日表兜底：被否，用户明确不维护静态表。
- 手动导入节假日 CSV：被否，用户明确不要。
- 多 API 源备选：被否，备选 API 也不可靠，增加复杂度。
- API 失败即报错终止：被否，单点依赖不可接受，工具应始终可用。
