# ADR-0002 工时生成核心策略

> 固化工时生成器的五项核心策略：比例化投入目标、员工轮转填充、年假随机分散、节假日 API 获取、不要求复现。

## 状态

accepted

## 上下文（Context）

- 员工有两个独立维度：**工种（job type）** 与 **业务线（business line）**。
- 一个项目要求**至少 1 名每个工种**的人员参与。
- 员工允许同时在多个项目上投入。
- 投入目标不是绝对工时，而是**比例**。
- 节假日数据来源：`http://timor.tech/api/holiday/year/xxxx` 在线 API。
- 合规规则固化（每日 ≤ 8h、节假日休假、年假休满）。
- 用途为回溯合规，不需要审计复现。

## 决策（Decision）

### D1 投入目标按比例表达

- 项目录入 `target_ratio`（如 0.3 = 30%），不录入绝对工时。
- 换算口径（待最终确认，当前推断）：项目应分配工时 = 该项目关联员工的可用工作日总容量 × `target_ratio`。
- 比例不要求跨项目求和为 100%；各项目独立表达投入强度。
- 开放问题：比例是相对"该项目关联员工"还是"全局员工池"？→ 当前推断为前者，待业务确认。

### D2 工时填充：员工轮转 + 项目持续性

- 按员工轮转填充，但保证员工在某个项目上的**持续性**——一旦开始投入某项目，不轻易切换到其他项目。
- 允许同一员工在多个项目投入：持续性是"单项目内连续"，不禁止跨项目并行。
- 轮转粒度：以连续工作日块为单位分配，而非逐日跳转。

### D3 年假安排：随机 + 倾向分散

- 年假在员工可投入区间内、非节假日的日期中随机选取。
- 倾向分散：避免连续大段年假，尽量打散到区间内多个片段。
- 不要求复现，使用随机种子不固定。

### D4 节假日数据：在线 API 获取 + 本地缓存

- 首次生成时调用 `http://timor.tech/api/holiday/year/xxxx` 按年度获取节假日。
- 获取结果本地缓存，后续生成优先读缓存；缓存缺失或年度未覆盖时回退到 API。
- 与 NFR-030（离线运行）的张力：纯离线场景下只能用已缓存年度的数据；未缓存年度在离线时报错而非静默用错数据。
- 调休工作日（周末补班）按 API 返回的 `workday` 标记处理。

### D5 不要求复现

- 不使用固定随机种子；同输入多次生成结果可不同。
- 理由：用途为回溯合规生成，非审计复现；放弃复现换取更自然的分布。

## 后果（Consequences）

- 正面：
  - 比例化目标更贴近真实项目投入表达方式。
  - 员工持续性约束让生成记录更像真实投入，降低"机器生成"特征。
  - 年假分散更符合常见休假模式。
  - 节假日 API 保证数据准确与年度更新，无需手维护。
- 负面：
  - 比例语义未完全钉死（D1 开放问题），实现前需确认。
  - 在线 API 引入网络依赖与不可用风险，离线场景受限。
  - 不复现导致 bug 难以稳定复现；测试需用固定种子局部注入确定性。
  - 持续性约束 + 多项目并行的填充算法复杂度上升，需谨慎设计避免死锁/回溯爆炸。
- 后续需要做的事：
  - 确认 D1 比例口径（关联员工 vs 全局员工池）。
  - 设计"员工轮转 + 持续性"填充算法的伪代码与边界用例。
  - 实现节假日 API 客户端 + 缓存层 + 离线降级策略。

## 备选方案（Alternatives）

- 投入目标用绝对工时：被否，用户明确为比例。
- 工时填充按日历顺序：被否，无法体现员工持续性，记录过于整齐。
- 年假集中安排：被否，倾向分散更自然。
- 节假日数据内置硬编码：被否，年度更新维护成本高，API 更准确。
- 固定种子保证复现：被否，用户明确不需要复现。
